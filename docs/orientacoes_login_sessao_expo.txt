ORIENTAÇÕES - INVENTÁRIO MOBILE | APLICAÇÃO EXPO
LOGIN, SESSÃO E COMUNICAÇÃO COM API LARAVEL

1. CONTEXTO GERAL

Este arquivo define as orientações para construção da parte de login e sessão da aplicação mobile do Inventário Mobile, desenvolvida em Expo/React Native.

A aplicação Expo será responsável pela interface mobile, captura das credenciais do usuário, envio dessas credenciais para a API Laravel e armazenamento seguro dos dados de sessão retornados pelo backend.

A API Laravel será exposta através do endpoint base:

https://stalling-saunch-wrecking.ngrok-free.dev/mobile-api/

Internamente, essa URL do ngrok estará redirecionando para o Laravel rodando localmente em:

http://127.0.0.1:8000/mobile-api/

Neste primeiro momento, o foco será exclusivamente na construção do fluxo de login e sessão.


2. OBJETIVO DO LOGIN MOBILE

O objetivo do login é permitir que o usuário do sistema EGap acesse a aplicação mobile utilizando as mesmas credenciais já utilizadas no sistema atual/legado.

Após o login ser validado pelo Laravel, a aplicação Expo deverá armazenar em sessão os seguintes dados do usuário autenticado:

- id;
- name;
- username;
- email;
- lotacao.

Essas informações serão utilizadas posteriormente para filtrar os bens patrimoniais vinculados à lotação/setor do usuário logado.


3. REFERÊNCIA DO SISTEMA ANTIGO

Nos dois projetos existe um arquivo do sistema antigo utilizado para login.

Esse arquivo deve ser usado como referência para entender:

- quais campos eram usados no login;
- como o usuário era localizado no banco;
- como a senha era validada;
- quais dados eram salvos na sessão;
- qual informação representa a lotação/setor do usuário;
- quais regras antigas precisam ser preservadas.

A implementação nova não precisa copiar o código antigo diretamente, mas deve respeitar a regra de negócio existente.

O fluxo antigo será usado como base conceitual, enquanto a nova implementação deverá seguir boas práticas com Expo, TypeScript, serviços de API e gerenciamento de sessão.


4. RESPONSABILIDADES DA APLICAÇÃO EXPO

A aplicação Expo será responsável por:

- criar a tela de login;
- capturar username/login e senha;
- validar campos obrigatórios antes de enviar a requisição;
- enviar as credenciais para a API Laravel;
- receber a resposta do backend;
- armazenar os dados da sessão localmente;
- controlar se o usuário está autenticado ou não;
- redirecionar o usuário para a área interna após login bem-sucedido;
- permitir logout removendo os dados da sessão;
- tratar mensagens de erro vindas da API.

A aplicação Expo não deve conter regra de autenticação sensível.

Validação de senha, busca de usuário, conferência de vínculo com lotação e regras de acesso devem ficar no Laravel.


5. ENDPOINT DE LOGIN A SER CONSUMIDO

O front deverá consumir o seguinte endpoint:

POST https://stalling-saunch-wrecking.ngrok-free.dev/mobile-api/login

O corpo da requisição deverá seguir um padrão simples, por exemplo:

{
    "username": "usuario",
    "password": "senha"
}

Caso o sistema antigo utilize outro nome de campo para login, como "login" em vez de "username", alinhar com o backend e manter uma única nomenclatura no contrato da API.

Recomendação inicial:

- No Expo, usar username;
- No Laravel, aceitar username e, se necessário, mapear internamente para o campo real da tabela antiga.


6. RESPOSTA ESPERADA EM CASO DE SUCESSO

Ao realizar login com sucesso, a API deverá retornar um JSON contendo os dados necessários para a sessão.

Formato esperado:

{
    "success": true,
    "message": "Login realizado com sucesso.",
    "user": {
        "id": 1,
        "name": "Nome do Usuário",
        "username": "usuario",
        "email": "usuario@email.com",
        "lotacao": {
            "id": 10,
            "nome": "Setor/Lotação do Usuário"
        }
    },
    "token": "token_opcional_caso_seja_usado"
}

Neste momento, se o backend ainda não utilizar token, o Expo pode salvar apenas o objeto user em sessão local.

Porém, a estrutura deve ser preparada para suportar token futuramente, caso seja adotado Sanctum, JWT ou outro mecanismo de autenticação.


7. RESPOSTA ESPERADA EM CASO DE ERRO

Em caso de erro, a API deverá retornar uma resposta padronizada.

Exemplo para credenciais inválidas:

{
    "success": false,
    "message": "Usuário ou senha inválidos."
}

Exemplo para usuário sem lotação:

{
    "success": false,
    "message": "Usuário não possui lotação vinculada."
}

Exemplo para erro de validação:

{
    "success": false,
    "message": "Informe usuário e senha para continuar.",
    "errors": {
        "username": ["O campo usuário é obrigatório."],
        "password": ["O campo senha é obrigatório."]
    }
}

O Expo deve exibir a mensagem principal presente no campo message.


8. ARMAZENAMENTO DA SESSÃO NO EXPO

Após login bem-sucedido, armazenar localmente os dados necessários da sessão.

Dados mínimos a serem salvos:

{
    "id": 1,
    "name": "Nome do Usuário",
    "username": "usuario",
    "email": "usuario@email.com",
    "lotacao": {
        "id": 10,
        "nome": "Setor/Lotação do Usuário"
    }
}

Recomendações:

- usar AsyncStorage ou Expo SecureStore;
- preferir SecureStore caso futuramente seja salvo token sensível;
- centralizar o controle de sessão em um AuthContext;
- criar métodos como login(), logout(), loadSession() e isAuthenticated;
- evitar espalhar acesso direto ao storage em várias telas.

Estrutura recomendada:

src/
  services/
    api.ts
    authService.ts
  contexts/
    AuthContext.tsx
  screens/
    LoginScreen.tsx
    HomeScreen.tsx
  types/
    auth.ts


9. SERVIÇO DE API NO EXPO

Criar um arquivo central para configuração da API.

Exemplo conceitual:

baseURL = "https://stalling-saunch-wrecking.ngrok-free.dev/mobile-api"

Todas as chamadas da aplicação mobile devem passar por esse serviço.

Não deixar URLs espalhadas diretamente em telas ou componentes.

O serviço de autenticação deve ter uma função semelhante a:

login(username: string, password: string)

Essa função será responsável por chamar:

POST /login

E retornar os dados já tratados para o AuthContext.


10. FLUXO ESPERADO NO FRONTEND

O fluxo de login na aplicação Expo deverá ser:

1. Usuário abre o aplicativo.
2. App verifica se existe sessão salva localmente.
3. Se existir sessão válida, redireciona para a área interna.
4. Se não existir sessão, mostra a tela de login.
5. Usuário informa username e senha.
6. Expo valida se os campos foram preenchidos.
7. Expo envia POST para /mobile-api/login.
8. Laravel valida as credenciais.
9. Laravel retorna dados da sessão.
10. Expo salva os dados localmente.
11. Expo atualiza o AuthContext.
12. Usuário é redirecionado para a tela inicial do inventário.


11. CONTROLE DE LOGOUT

O logout deverá:

- remover os dados locais da sessão;
- limpar o AuthContext;
- redirecionar o usuário para a tela de login.

Mesmo que a autenticação inicial não utilize token, o logout local já deve existir desde o começo.


12. CUIDADOS IMPORTANTES NO EXPO

- Não validar senha no front;
- Não salvar senha localmente;
- Não hardcodar usuário de teste em produção;
- Não espalhar a URL da API em várias telas;
- Não assumir que o usuário sempre terá lotação;
- Tratar falha de conexão com o ngrok;
- Tratar erro 401 como sessão inválida;
- Tratar erro 422 como erro de validação;
- Exibir loading durante a tentativa de login;
- Bloquear múltiplos cliques no botão enquanto a requisição estiver em andamento.


13. OBSERVAÇÃO SOBRE NGROK

Durante o desenvolvimento, o Expo deverá consumir a API através de:

https://stalling-saunch-wrecking.ngrok-free.dev/mobile-api/

Esse endereço aponta para o backend Laravel local.

Caso o domínio do ngrok mude, atualizar apenas o arquivo central de configuração da API no Expo.


14. RESULTADO ESPERADO DESTA ETAPA

Ao final desta etapa, a aplicação Expo deverá possuir:

- tela de login funcional;
- integração com o endpoint de login do Laravel;
- armazenamento local da sessão;
- AuthContext configurado;
- redirecionamento baseado em autenticação;
- logout funcional;
- tratamento básico de erros;
- sessão contendo id, name, username, email e lotacao.
